Lesson 8: Advanced Classes

Class Variables

Class variables are shared amongst all instances of a class. Recall that instance variables are unique to individual objects created from the class.

To clarify, given a class MyClass,

class MyClass:
    def __init__(self, foo):
        self.foo = foo

an instance my_class of the class is created by invoking the constructor of that class.

>>> my_class = MyClass(42)

Now we can access the instance variable foo of my_class.

>>> my_class.foo
42

Notice that we can't access the my_var variable from the class itself.

>>> MyClass.foo
AttributeError: type object 'MyClass' has no attribute 'foo'

The key here is that my_var lives in the namespace of the my_class instance and not the type object MyClass itself, which we can easily verify.

>>> 'foo' in my_class.__dict__
True
>>> 'foo' in MyClass.__dict__
False

Notice that instance variables are defined on the self parameter inside the __init__ constructor. Since class variables belong to the type object itself, they are defined as variables directly in the class's scope. That is, they not defined inside the constructor because they have nothing to do with the particular instances being created.

class MyClass:
    bar = 'Hello World!'

    def __init__(self, foo):
        self.foo = foo

Now, when we create multiple instances, they will all share the same value of bar, but they will each have their own foo.

>>> instance_1 = MyClass(42)
>>> instance_2 = MyClass(36)

>>> instance_1.foo
42
>>> instance_2.foo
36
>>> instance_1.bar
'Hello World!'
>>> instance_2.bar
'Hello World!'

We can also access bar directly from the type object MyClass.

>>> MyClass.bar
'Hello World!'

Now, let's verify that foo only lives in the instance's namespace, while bar only lives in the type object's namespace.

>>> 'foo' in my_class.__dict__
True
>>> 'bar' in my_class.__dict__
False

>>> 'foo' in MyClass.__dict__
False
>>> 'bar' in MyClass.__dict__
True

Okay, one last piece of puzzle. What happens if I change the value of bar on the type object MyClass?

>>> MyClass.bar = 'Something different'

>>> instance_1.bar
'Something different'
>>> instance_2.bar
'Something different'

Naturally, the value is updated for all of the instances because bar exists at the class level. But what happens if I try to modify bar on only one of the instances?

>>> instance_1.bar = 'Unique to me!'

>>> instance_1.bar
'Unique to me!'
>>> instance_2.bar
'Something different'

Rather than updating the value for all instances, it only modified that one instance. To understand what's going on, let's check the namespaces again.

>>> 'foo' in my_class.__dict__
True
>>> 'bar' in my_class.__dict__
True

>>> 'foo' in MyClass.__dict__
False
>>> 'bar' in MyClass.__dict__
True

Notice that bar is now a contained in the namespace of instance_1. So we actually didn't modify the class variable, we created an instance variable. This is because Python allows the creation of instance variables via attribute assignment.

>>> instance_1.baz = 'Happy Birthday'

>>> 'baz' in instance_1.baz
True
>>> instance_1.baz
'Happy Birthday'

Notice that even though baz was never declared in the class definition, Python let me assign it a value.

When we attempt to access attributes, Python checks a number of locations in a specific order to see if exists:

  1. the namespace of the instance
  2. the namespace of the type object
  3. the namespaces of any type objects the class inherits from

We'll talk about inheritance structures in just a bit, so don't worry about that for now. The point is, Python first investigates the instance, then the type object. So when we tried to change the value of bar on instance_1, we ended up creating a instance variables that superseded the class variable when we tried to access the attribute.

Understanding this distinction is key because we can see different behavior in our methods if we use MyClass.bar instead of self.bar. Using the MyClass.bar guarantees that the class variables is used, while self.bar will use an instance variable if it exits, otherwise it will fall back to the class variable.

For more information, check out the docs.

Class methods

Just like variables, methods can act on instances of the class or the class itself. Instance methods are the standard functions we've seen in classes up to this point. Recall that instance methods always start with the a special parameter (usually called self) that refers to the particular instance that calls that function.

class MyClass:
    def __init__(self, x, y):
        self.x = x
        self.y = y
    
    def my_instance_method(self):
        print(self.x, self.y)
        
>>> my_instance = MyClass('a', 'b')
>>> my_instance.my_instance_method()
a b

This is equivalent to calling the method directly on the class object and passing the instance as the first argument.

>>> MyClass.my_instance_method(my_instance)
a b

Now, don't be fooled, this is still an instance method because we need to pass the instance as argument.

However, there is a way to create class methods which act on the class object directly. We achieve this by way of the classmethod decorator.

class MyClass:
    z = 'c'

    @classmethod
    def my_class_method(cls):
        print(cls.z)
>>> MyClass.z
'c'
>>> MyClass.my_class_method()
c

Much like instance methods, class methods take an implicit first argument (typically called cls to mean class, since class is restricted keyword in Python), but rather than referencing a particular object, it refers to the type object itself.

While it is possible to call class methods from instance variables, it isn't common in best practices.

Static Methods

There is third possibility when it comes to methods. Sometimes we have logic that is related to the class, but doesn't affect any instances or even the class itself. That is, the method performs some static calculations that are relevant to the class. These are called static methods, and they are accessed with the staticmethod decorator.

Unlike instance and class methods, static methods have no implicit first argument since they don't act on anything (this is why they are called static).

class MyClass:
    @staticmethod
    def my_static_method():
        print(2 + 2)
>>> MyClass.my_static_method()
4

>>> instance = MyClass()
>>> instance.my_static_method()
4

Notice that static methods can be called from either instances or the type object directly.

Inheritance

When we have classes that share a lot of the same functionality, we can create a hierarchical model in which on class become the child of another class. This parent-child relationship is meant to evoke the idea that the child inherits the traits of its parent.

class Employee:
    raise_amount = 1.04
    
    def __init__(self, first_name, last_name, pay):
        self.first_name = first_name
        self.last_name = last_name
        self.email = f"{self.last_name}.{self.first_name}@company.com"
        self.pay = pay
    
    def apply_raise(self):
        self.pay *= self.raise_amount
>>> employee = Employee('John', 'Doe', 60000)
>>> employee.email
'Doe.John@company.com'
class Developer(Employee):
    pass
>>> developer = Developer('Jane', 'Doe', 70000)
>>> developer.email
'Doe.John@company.com'

Notice that for all intents and purposes, developer behave exactly like employee even though we haven't added any code to the Developer class. Now, if we decide that we want to reward our developers a little more than other generic employees, we could simply change the raise_amount for the Developer class.

class Developer(Employee):
    raise_amount = 1.07
>>> employee.apply_raise()
>>> employee.pay
>>> developer.apply_raise()
>>> developer.pay

The parent class is called a superclass and the child class is called a subclass. If we want our subclasses to be initialized with additional variables we can override the class initializer. We still want to use the constructor logic from the superclass, so Python give us access to the super keyword as an alias for the superclass.

class Developer(Employee):
    raise_amount = 1.07
    
    def __init__(self, first_name, last_name, pay, primary_language):
        super().__init__(first_name, last_name, pay)
        self.primary_language = primary_language

We may also want our subclasses to have additional functionality, or even override existing functionality. If we define a method with the same signature as a superclass, we will essentially replace the default behavior with our own.

class Manager(Employee):
    raise_amount = 1.10
    bonus = 2000
    
    def __init__(self, first_name, last_name, pay, employee):
